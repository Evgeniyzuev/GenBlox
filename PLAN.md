# План-концепт GenBlox

## 1. Цель

GenBlox — каталог небольших браузерных игр («спейсов»), в которые можно играть
одному или вместе с участниками одной комнаты.

All public-facing website copy, interface text, game names, game rules, status
messages, HUD labels, and errors must be written in English.

Комната не принадлежит конкретной карточке игры. Пользователь сначала создаёт
или находит комнату, после чего вся группа свободно выбирает игры из общего
каталога.

## 2. Основные сущности

### Каталог

Главный экран со всеми доступными спейсами. Каталог доступен как пользователю
без комнаты, так и участникам комнаты.

### Комната

Временная группа пользователей PlayroomKit. Комната хранит:

- код и ссылку-приглашение;
- список участников и хозяина;
- режим доступа: открытый или приватный;
- текущий экран группы: каталог или игра;
- идентификатор активной игры;
- состояние запущенной игры.

### Игра

Независимый модуль с правилами и интерфейсом. Игра не создаёт комнату и не знает,
как пользователи в неё вошли. Контроллер комнаты передаёт игре участников и
синхронизирует её состояние.

## 3. Главное меню

Меню комнаты располагается отдельно от карточек игр и доступно в шапке сайта.

Состояние без комнаты:

- **Играть одному** находится на карточке конкретной игры;
- **Найти открытую комнату** находится в меню комнат;
- **Создать комнату** находится в меню комнат;
- **Войти по коду или QR** находится в меню комнат.

Состояние внутри комнаты:

- код комнаты;
- количество и список участников;
- кнопка копирования ссылки;
- кнопка показа QR;
- кнопка выхода;
- признак открытой или приватной комнаты.

Карточка игры внутри комнаты вместо создания новой комнаты показывает кнопку
**«Запустить для комнаты»**.

## 4. Одиночный режим

1. Пользователь без комнаты выбирает карточку игры.
2. Нажимает **«Играть одному»**.
3. Игра запускается локально, без PlayroomKit.
4. Если игре нужен соперник, используется компьютер.
5. Закрытие игры возвращает пользователя в каталог.

Одиночная партия не должна случайно создавать комнату или публиковаться в
матчмейкинге.

## 5. Создание комнаты

### Открытая

1. Пользователь открывает независимое меню комнат.
2. Нажимает **«Найти открытую комнату»**.
3. PlayroomKit ищет открытую комнату со свободным местом.
4. Если она найдена, пользователь подключается к ней.
5. Если за время ожидания подходящая комната не найдена, PlayroomKit создаёт
   новую открытую комнату.
6. Пользователь попадает в общий каталог и ждёт других участников.

Таким образом, отдельный публичный список комнат на сайте не требуется:
матчмейкинг сам выбирает подходящую комнату.

### Приватная

1. Пользователь нажимает **«Создать приватную комнату»**.
2. PlayroomKit создаёт комнату, не участвующую в открытом поиске.
3. В меню комнаты отображаются код, ссылка и QR.
4. Хозяин отправляет приглашение другим пользователям.

## 6. Подключение к существующей комнате

### Через открытый поиск

Оба пользователя нажимают **«Найти открытую комнату»**. Второй пользователь
автоматически подключается к первой совместимой комнате со свободным местом.

### Через код

1. Пользователь открывает меню комнат.
2. Выбирает **«Войти по коду или QR»**.
3. Вводит код комнаты.
4. Код нормализуется: удаляются пробелы, буквы переводятся в верхний регистр.
5. После проверки пользователь подключается к комнате.
6. При неверном, закрытом или переполненном коде показывается понятная ошибка,
   а введённое значение не стирается.

### Через ссылку

Ссылка содержит идентификатор комнаты, например:

```text
https://genblox.example/#r=ABCD
```

При открытии ссылки сайт:

1. читает код из URL;
2. показывает подтверждение входа;
3. после подтверждения подключается к комнате;
4. удаляет временные параметры из истории после успешного входа.

### Через QR с камеры

1. Пользователь выбирает **«Сканировать QR»**.
2. Сайт запрашивает доступ к камере только после нажатия кнопки.
3. QR распознаётся через `BarcodeDetector`, если он поддерживается браузером.
4. Для неподдерживаемых браузеров используется небольшой fallback-декодер.
5. Из результата принимается только ссылка GenBlox или корректный код комнаты.
6. Камера немедленно выключается после распознавания, отмены или закрытия меню.

Отказ в доступе к камере не блокирует вход: остаются ввод кода и загрузка
изображения.

### Через изображение QR

1. Пользователь нажимает **«Загрузить изображение QR»**.
2. Выбирает PNG, JPEG или WebP.
3. Изображение декодируется локально в браузере и никуда не загружается.
4. После успешного распознавания показывается найденный код и подтверждение
   подключения.
5. Если QR не найден, пользователь может выбрать другое изображение или ввести
   код вручную.

Для MVP достаточно QR, создаваемого PlayroomKit, и сканирования обычной камерой
телефона. Встроенные камера и загрузка изображения являются следующим этапом.

## 7. Работа комнаты как общего хаба

После подключения пользователь остаётся в комнате при переходах по каталогу.

Синхронизируемое состояние комнаты:

```js
{
  screen: "catalog",          // catalog | game
  activeGame: null,           // например tic-tac-toe
  startedBy: null,
  revision: 1
}
```

Запуск игры:

1. Любой участник нажимает **«Запустить для комнаты»**.
2. Контроллер создаёт начальное состояние выбранной игры.
3. В комнате устанавливается `activeGame`.
4. Игра автоматически открывается у всех участников.
5. Подключившийся позже игрок получает текущее состояние и открывает уже
   запущенную игру.

Чтобы два участника не запустили разные игры одновременно, команда запуска
содержит номер ревизии. При конфликте принимается первая подтверждённая команда,
остальным показывается уже выбранная игра.

## 8. Завершение игры и возврат в каталог

- **Закрыть игру** означает вернуть всю комнату в каталог.
- Перед групповым закрытием показывается подтверждение:
  «Завершить игру для всех участников?».
- Обычный участник может выбрать **«Скрыть только у меня»**, но при повторной
  синхронизации ему предлагается вернуться в активную игру.
- Завершённое состояние партии очищается только после перехода комнаты в каталог.
- Комната и её участники при этом сохраняются.

## 9. Выход из комнаты

Кнопка **«Выйти из комнаты»** находится только в независимом меню комнаты.

Последовательность:

1. Если запущена игра, пользователь получает предупреждение.
2. Соединение PlayroomKit закрывается.
3. Локальное состояние комнаты и игроков очищается.
4. Параметр комнаты удаляется из URL.
5. Интерфейс возвращается в каталог без комнаты.

Если выходит хозяин:

- роль хозяина передаётся другому участнику через `transferHost`;
- активная игра продолжается;
- если других участников нет, временная комната прекращает существование.

Потеря сети не считается немедленным добровольным выходом. Сначала показывается
состояние **«Переподключение…»**, затем выполняется ограниченное число попыток.

## 10. Назад, закрытие меню и история браузера

Интерфейс рассматривается как набор экранов:

```text
catalog
room-menu
join-menu
qr-scanner
game
confirm-exit
```

Правила навигации:

- кнопка **×** закрывает только верхнее меню;
- **Escape** работает так же, как кнопка ×;
- кнопка браузера **Назад** сначала закрывает верхнее меню;
- из игры кнопка **Назад** открывает подтверждение выхода из игры;
- из каталога кнопка **Назад** работает как обычная история браузера;
- закрытие меню подключения отменяет поиск и выключает камеру;
- закрытие меню комнаты не означает выход из комнаты;
- выход выполняется только отдельной кнопкой с подтверждением;
- повторное быстрое нажатие не создаёт несколько комнат и подключений.

Каждое открытое полноэкранное меню добавляет запись через `history.pushState`.
Обработчик `popstate` закрывает верхний экран, не перезагружая страницу.

## 11. Ошибки и пограничные случаи

Должны быть обработаны:

- комната не найдена;
- комната заполнена;
- хозяин вышел;
- соединение потеряно;
- матчмейкинг отменён пользователем;
- PlayroomKit не загрузился;
- камера запрещена или отсутствует;
- изображение не содержит QR;
- QR ведёт на другой сайт;
- одновременно запущены две игры;
- второй игрок отключился во время хода;
- пользователь открыл приглашение, уже находясь в другой комнате.

Ошибки отображаются внутри соответствующего меню, а не через `alert`.

## 12. Архитектура

```text
app-shell
├── navigation-controller       история, назад, модальные экраны
├── room-controller             жизненный цикл комнаты
│   └── playroom-client         адаптер PlayroomKit
├── invite-controller           код, ссылка, камера, изображение QR
├── game-registry               список независимых спейсов
└── games
    ├── tic-tac-toe             правила и локальный компьютер
    └── checkers                русские шашки и локальный компьютер
```

PlayroomKit используется только через `playroom-client`. Игры не вызывают SDK
напрямую. Это позволяет тестировать правила локально и позже заменить сетевой
слой без переписывания игр.

## 14. Критерии готовности

- Комнату можно создать независимо от выбора игры.
- Два пользователя соединяются через поиск, код, ссылку или QR.
- Любой участник запускает игру у всей комнаты.
- Возврат в каталог не разрушает комнату.
- Выход из комнаты выполняется явно и корректно передаёт роль хозяина.
- Назад, Escape и × предсказуемо закрывают только текущий экран.
- Камера всегда выключается после закрытия QR-сканера.
- Одиночный режим не зависит от сети.

## 15. GenBlox Creator

### Пользовательский путь

Ребёнок открывает **Create a Game**, выбирает компактный шаблон, скачивает один файл
`*.genblox.txt`, прикладывает его к любому AI-чату, описывает изменения и загружает
полный изменённый файл обратно. На сайте доступны тот же путь через копирование и
вставку текста, безопасный предпросмотр, понятные ошибки и история локальных версий.

Большие встроенные игры не экспортируют производственный исходный код. Действие
**Make your own version** открывает компактный самостоятельный ремикс-шаблон.

### Переносимый формат

```text
=== GENBLOX GAME 1 ===
=== MANIFEST ===
{ ...JSON... }
=== HTML ===
...
=== CSS ===
...
=== JAVASCRIPT ===
...
=== END ===
```

Манифест содержит `formatVersion`, `templateId`, `title`, `description`, `mode`,
`maxPlayers`, `orientation` и `sdkVersion`. Загрузка файла и вставка текста проходят
через один parser, а скачивание и копирование — через один serializer.

### Изоляция и SDK

Пользовательский HTML/CSS/JavaScript выполняется только в `iframe` с `sandbox="allow-scripts"`
и строгой CSP без внешней сети. Игра не получает DOM, cookies, localStorage или профиль
основного сайта. SDK v1 предоставляет `ready`, безопасный профиль-заглушку, `finish` и
`restart`; сообщения принимаются только от iframe предпросмотра. NPM, CDN и серверный
код в пользовательских играх не поддерживаются.

### Версии, тестирование и публикация

MVP хранит до 10 неизменяемых черновых версий локально на устройстве. Первый online-этап
добавляет passwordless-вход через email, приватные неизменяемые версии в Supabase и
экран **My Games** для открытия либо удаления online-проекта. Следующий этап добавит
приватные тестовые ссылки друзьям.
Публичный каталог включается только после изоляции, аккаунтов и модерации; каждая новая
версия проверяется отдельно.

### Поток публикации

Публикация разделена на два независимых уровня. Сначала автор входит через email,
сохраняет приватный online-черновик и получает неизменяемую версию. Затем
появятся секретные ссылки для тестирования с друзьями. Только явно выбранная версия
может быть отправлена на автоматическую проверку и ручную модерацию; после одобрения
она становится доступной в общем каталоге. Новая версия не заменяет уже опубликованную.

Основной сайт хранит аккаунт и метаданные, а опубликованные builds в будущем запускаются
с отдельного игрового домена без cookies основного сайта. Исходный TXT остаётся закрытым,
пока автор явно не разрешит ремиксы. Публичные комментарии, личные сообщения и загрузка
фотографий не входят в детский MVP.

Серверные статусы версии: `private_draft`, `private_test`, `review`, `published`,
`rejected`, `disabled`. Доступ к приватным строкам ограничивается Supabase RLS по
`auth.uid()`; service-role/JWT/Postgres-секреты никогда не передаются браузеру.

### Обучение

Первый запуск показывает пропускаемый тур из семи простых шагов от выбора шаблона до
тестирования. Постоянная кнопка **How to make a game** открывает его повторно. Интерфейс
и подсказки написаны простым английским для детей 8–12 лет, содержат готовый AI-промпт,
следующее действие после каждой ошибки и памятку не сообщать настоящее имя, адрес,
школу, пароль и чужие фотографии.
